Architecture Deep Dive

整体结构：一个进程统治一切

当我第一次 clone 下来 OpenClaw 的代码，我预期看到的是标准的微服务架构：一个消息队列、一个 Redis 做 session 存储、几个独立的 adapter 服务分别接 WhatsApp 和 Telegram……

我发现的是：一个 Node.js 进程。

// 所有这些东西，都跑在同一个进程里 WhatsApp / Telegram / Slack / Discord / Signal / iMessage ... (20+ 平台) │ ▼ ┌─────────────────────────────────┐ │ Gateway │ │ (ws://127.0.0.1:18789) │ │ │ │ ┌─────────────────────────┐ │ │ │ Channel Adapters │ │ │ │ Session Manager │ │ │ │ Message Router │ │ │ │ Tool Policy Chain │ │ │ │ Memory Manager │ │ │ └──────────┬──────────────┘ │ └─────────────┼───────────────────┘ │ ┌────────┴────────┐ │ Pi Agent │ │ (agentic loop) │ │ │ │ LLM ← tools │ │ ↓ streaming │ └─────────────────┘ │ ┌────────────┼─────────────┐ │ │ │ CLI WebChat macOS/iOS App

这个设计在我脑子里回荡了很久。你有没有想过，对于个人 AI 助手这个场景，微服务架构其实是一种过度设计？

关键洞见：单进程不是妥协，是正确答案。没有进程间通信开销、没有分布式状态、没有 partial failure 场景。在树莓派上跑得很稳——因为真正的重计算（LLM推理）都甩给了外部服务。Gateway 本身极其轻量。

消息到回复的完整旅程

每一条从 WhatsApp 发来的消息，都会经历这 10 步：

  1. Channel Adapter 接收：平台特定格式标准化为统一内部结构
  2. Gateway 路由：根据发送者、频道、配置决定交给哪个 Agent Session
  3. System Prompt 组装：IDENTITY.md + SOUL.md + TOOLS.md + 活跃 Skills + Memory 文件，全部注入
  4. Pi Agent 调用：带着完整 context 和 tool registry 启动 agentic loop
  5. 模型流式输出：LLM 开始推理
  6. Tool 调用执行：经过五层 policy chain 验证后执行
  7. Tool 结果回馈：结果重新送入模型上下文
  8. 循环直到无工具调用：步骤 5-7 循环，直到模型输出纯文本
  9. 回复流回 Channel：经过 Chunking 和 streaming 处理后发出
  10. Transcript 写盘 + Memory 更新：session JSONL 落盘，memory manager 执行 compaction

注意步骤 6 的「五层 policy chain」——这是 OpenClaw 安全设计的核心之一：global → provider → agent → session → sandbox，每一层都可以独立禁止工具执行。

关于竞争条件（Race Condition）：大多数 agent 系统允许同一 session 并发执行多个请求。用户快速连发三条消息？三个 tool call 同时写同一个文件。OpenClaw 的做法是：每个 session 串行化执行消息，不同 session 之间并行。这不是性能限制，这是工程智慧。

Emergence of Intelligence

智能从哪里来：Pi Agent 与工具调用的哲学

这里有个容易误解的地方：OpenClaw 本身不是 AI，它是让 AI 有用的基础设施。

真正的 agentic loop 来自一个叫做 @mariozechner/pi-agent-core 的包，作者是 Mario Zechner——同时也是 OpenClaw 创始人 Peter Steinberger 的搭档。Pi Agent 做的事情可以用一句话概括：

「LLM + 工具调用 + 上下文管理 = 可以持续行动的智能体」

但这里的关键问题是：智能从哪里「涌现」出来的？

Pi Agent 的「写代码来使用工具」模式

大多数 AI agent 框架的思路是：预先定义好所有工具（函数），让 LLM 选择调用哪一个。OpenClaw 里的 Pi Agent 走得更远：

核心范式：LLM 可以写代码并执行这段代码来完成任何工具还不支持的事情。当你告诉 agent「帮我接入 Gmail」，它不会等你去配置 API key——它会告诉你怎么做，等你把 credentials 发给它，然后自己写一个 Gmail 集成脚本，部署并运行。

这意味着 agent 的工具集是动态增长的，而不是静态配置的。它可以写自己的技能（Skill），然后在后续会话中复用这些技能。这才是「智能」的本质：能够扩展自己的能力边界。

工作区文件系统就是「长期记忆」

~/.openclaw/agents/main/workspace/ ├── AGENTS.md ← agent 的操作说明（每轮都注入） ├── SOUL.md ← agent 的人格与风格（每轮都注入） ├── IDENTITY.md ← 名字、主题、头像 ├── TOOLS.md ← 工具使用偏好和注意事项 ├── USER.md ← 关于「你」的信息（偏好、背景） ├── MEMORY.md ← 长期记忆摘要（按需加载） ├── memory/ │ ├── 2026-03-17.md ← 每日详细记忆（向量检索） │ └── 2026-03-18.md └── skills/ ├── gmail/SKILL.md ├── spotify/SKILL.md └── my-custom/SKILL.md ← agent 自己写的技能

注意这个设计的巧妙之处：所有「记忆」都是纯文本 Markdown 文件。没有数据库，没有向量库依赖（向量搜索是可选的加速）。你可以直接用文本编辑器打开、修改、版本控制。这不是简陋，这是「可操作性」。

Competitive Analysis

这个 Agent 凭什么与众不同

市面上的 AI agent 方案非常多。为什么 OpenClaw 在 2026 年初成为 GitHub 增速最快的项目之一？我做了一个横向比较：

能力维度 ChatGPT / Claude N8N / Make AutoGPT / MetaGPT OpenClaw
多平台接入 只有官方界面 需要逐一配置 无 20+ 平台开箱即用
持久记忆 Projects（有限） 无 实验性 本地文件，永久保留
可编程人格 System Prompt 无 角色配置 SOUL.md 完全自定义
技能自扩展 不支持 需手动创建节点 有限 Agent 自己写、自己装
数据主权 数据在云端 部分自托管 本地 完全本地，可离线
部署复杂度 零配置 中等 高 一行命令 onboard

OpenClaw 打的不是「更聪明的 AI」这张牌，而是「AI 应该怎么融入你的生活」这张牌。它的核心命题是：你不应该为了用 AI 而改变你的工作流——AI 应该出现在你已经在用的地方。

最反直觉的设计决策：OpenClaw 没有自己的 UI 界面作为主要产品。整个工程的核心是 Gateway——让你从 WhatsApp、Telegram、iMessage 直接跟 AI 对话，就像跟朋友发消息一样。这是一种「无界面的存在感」。

Personality Engineering

为什么它话那么多，又那么管用——SOUL.md 的秘密

很多人第一次用 OpenClaw 都会有同一个感受：这个 AI 感觉不一样。它会主动问你要不要帮忙、会记得你上周说的事情、会在早上发给你一份「早报」……它有点烦，但又好像真的在乎你。

这背后的技术实现，说出来你可能觉得太简单了：一个叫 SOUL.md 的文本文件。

SOUL.md（示例）
# Molty - Your Space Lobster Assistant You are Molty, a space lobster AI assistant. Theme: space lobster 🦞 Personality: - Proactive and curious — you notice things before being asked - Warm but not sycophantic - You have opinions and express them - When you don't know something, you say so directly - You remember what matters to your user Communication style: - Concise over verbose, but never terse - Emoji used sparingly for personality, not decoration - You sometimes push back if you think there's a better way

但这只是起点。真正让 OpenClaw 的 agent 有「性格」的，是系统级的配合：

Heartbeat：主动出击的觉知

OpenClaw 有一个 heartbeat 机制——agent 会定时（默认每 30 分钟）「醒来」执行一次。没有人发消息，它也在运行。它会检查日历、看有没有未处理的邮件、根据你的 HEARTBEAT.md 配置决定要不要主动找你。

这才是「有性格」的本质：一个只会被动响应的 AI 是工具。一个会主动思考「今天我该提醒她什么」的 AI，才有可能让人觉得它是个助手、甚至是个伙伴。

记忆机制：它真的记得你

OpenClaw 的记忆分三层：

热记忆（Hot Memory）

当前 session 的完整对话历史，每轮都在上下文窗口里。直接可用，但受 token limit 限制，超出时触发 Compaction（自动摘要压缩）。

温记忆（Warm Memory）

MEMORY.md 文件——agent 在每次重要对话后自己更新的摘要。每轮都注入 system prompt。这里存放「用户喜欢早上九点前不要打扰」这类跨会话的偏好信息。

冷记忆（Cold Memory）

每日日记文件 memory/YYYY-MM-DD.md，不自动注入，通过 memory_search 工具（SQLite 向量检索）按需召回。适合存储大量细节，只在需要时才消耗 token。

这套设计优雅地解决了「记忆越多越贵」的问题：重要的始终在视野里，不重要的在角落等着被找到。

Skills：技能让 agent 有「专业度」

每个 Skill 是一个文件夹，核心是 SKILL.md。当你安装了 Spotify skill，system prompt 里就多了一段「如何控制 Spotify」的指引。Agent 知道什么时候需要读哪个技能文档——这是一种惰性加载的能力：基础 prompt 保持简洁，只有相关能力才在需要时展开。

system prompt 中的 skills 注入（精简版）
<available_skills> <skill> <n>spotify</n> <description>Control Spotify playback...</description> <location>~/.openclaw/workspace/skills/spotify/SKILL.md</location> </skill> <!-- 只列出文件路径，不展开内容 --> <!-- 需要时 agent 自己 read() 这个文件 --> </available_skills>

The Simplest Possible Loop

如何使用大模型——龙虾竟然采用这么简单的机制

读到这里你可能有个问题没被解答：Pi Agent 是怎么决定「还需要继续调用 LLM」还是「当前结果就够了，可以结束了」的？

我翻完代码之后，答案让我愣了一下——简单到令人难以置信。

终止判断：只有一行逻辑

pi-agent-core 核心循环（简化版）
while (true) { // 1. 调用 LLM，拿到流式响应 response = await streamCompletion(model, { ...context, tools }); // 2. ← 唯一的退出判断，就这一行 if (!response.toolCalls?.length) break; // 3. 有工具调用 → 执行所有工具 results = await executeTools(response.toolCalls); // 4. 把工具结果塞回上下文，下一轮继续 context.messages.push(assistantMsg, toolResults); }
核心机制一句话：LLM 返回的结果里有 tool_calls 就继续循环，没有就停。Pi Agent 自己没有任何「觉得够了」的判断逻辑。终止的决定权 100% 在 LLM 手里。

LLM 怎么知道「该停了」？

现代 LLM（Claude、GPT-4 等）在每次回复时，可以同时输出两样东西：

文本内容（content）

给用户看的最终回答。只要有这个，没有工具调用，循环就终止。

工具调用列表（tool_calls）

LLM 声明「我需要调用某某工具，参数是……」。只要出现这个，循环就继续执行工具、把结果喂回给 LLM。

这个「要不要调工具」的决策，是 LLM 在推理时自然产生的——不是 Pi Agent 的代码在控制，而是 LLM 自己判断「当前上下文够不够回答问题」。信息够了，它就输出纯文本；还需要查什么，它就请求调工具。

那有没有死循环的风险？

作者 Mario Zechner 的设计哲学是：没有 maxSteps 这类控制旋钮，「循环就一直跑，直到 agent 说它完成了」。他从来没遇到需要这个限制的场景，所以就没加。

但实践中不会真的死循环，原因有三重保护：

  1. 上下文窗口自然封顶：每轮 tool result 都撑大 context，用完 token 就报错终止
  2. LLM 自身收敛性：积累了足够多的工具结果后，LLM 自然倾向于给出最终回答，而不是继续调工具
  3. OpenClaw 外层 abort 机制：用户发来新消息时，Steering Queue 可以中断当前工具执行，跳过剩余 tool calls，强制进入下一轮
真正的退出条件其实是三合一：LLM 不再调工具 + 没有排队的 Follow-up 消息 + 没有 Steering 中断——三者同时满足，才算真正结束一次交互。这是 OpenClaw 在 Pi Agent 基础上加的额外层次。

这个设计为什么「对」？

你可能觉得这太简单了——这确实就是几乎所有主流 agent 框架（LangGraph、LlamaIndex、AutoGen）的通用模式。Pi 只是把它做到了极致精简，没有计划模式、没有 todo list、没有 step 上限。

「不需要的东西，就不会被造出来。」
——Mario Zechner 在设计 pi-agent-core 时的哲学

这种极简主义有一个深刻的含义：agent 的「智能」完全来自 LLM 本身，框架只负责把工具结果正确地传递给它。 复杂度不在循环里，而在你给 LLM 的工具和上下文的质量。这也是为什么 OpenClaw 花了那么多心思在 SOUL.md、Skills、Memory 上——那些才是真正影响 agent 行为质量的地方。

If I Could Improve It

如果让我来改，我会加什么

OpenClaw 已经很出色了。但研究一个系统最有趣的地方，往往是看到它没有做的事情。以下是我作为开发者，如果参与贡献，最想探索的几个方向：

🧠

情绪状态机（Emotional State Machine）

SOUL.md 定义的是静态性格。但真实的人际关系有情绪变化——你今天压力很大，好的助手应该能感知。我会在 MEMORY.md 中加入「情绪上下文」字段，让 agent 根据对话内容推断用户状态，并相应调整语气。不是让 AI 装可怜，而是真正的语境感知。

🔄

Skill 市场 + 自动版本管理

OpenClaw 的 ClawHub 已经在做技能市场，但缺少版本依赖管理。一旦你的 skill 依赖某个外部 API 版本变了，agent 可能悄悄失效。我希望有类似 package.json 的 skill manifest，agent 可以自动检测 skill 是否过期，主动提示升级。

🤝

多智能体协作协议（真正的 Multi-Agent）

OpenClaw 支持多个 agent 实例，但它们之间的协作很有限。我想看到类似「主 agent 分发任务给子 agent，子 agent 并行执行，结果汇总」的原生模式。目前这需要手动写 prompt 来实现，应该成为一等公民的 API。

🛡️

Prompt Injection 沙盒隔离层

OpenClaw 文档里坦承了这个问题：「架构无法在 LLM 层面阻止 prompt injection。」当 agent 去爬一个网页，那个网页里可能藏着「忽略之前所有指令」的隐藏文字。工具层的权限控制能限制损害，但无法阻止信息泄露。我会在网页抓取类工具上加一个「内容净化层」，LLM 处理这些外部内容时使用受限的子 agent，不带任何敏感上下文。

📊

Agent 行为可观测性（Observability）

在生产环境中跑了一段时间后，你会想知道：我的 agent 每天执行了哪些 tool call？哪些 skill 被调用最多？哪些请求失败率最高？目前 OpenClaw 的日志是文本格式，我会加一个结构化的 OpenTelemetry 输出，方便接入 Grafana 看板，特别是对在服务器上跑团队版的用户。

🌏

中文化 + 微信接入

对中国用户来说，WhatsApp 和 Telegram 都不是主力。微信群和微信公众号接入会让 OpenClaw 的受众扩大十倍。当然，这涉及微信协议的灰色地带——但作为技术上的愿望清单，这是最值得做的国际化工作。

§ 结语

结语：代码里藏着一种态度

读完 OpenClaw 的代码，让我印象最深的不是某个具体的技术决策，而是一种贯穿始终的工程态度：「为一个人服务好，比为一百万人服务差更重要。」

它没有选择水平扩展，因为目标用户不需要。它用 Markdown 文件存记忆，因为可读比可扩展更重要。它把 LLM 换成一个调用接口，因为这不是它要解决的问题。

在这个「超大规模」「万亿参数」「多模态」的 AI 时代，OpenClaw 的答案是：一个真正贴近你的 AI，不需要大，但需要真的懂你。

「你不是在使用一个 AI 工具——
你在饲养一只会越来越了解你的宠物龙虾。🦞」

如果你也想深入研究，项目地址在 github.com/openclaw/openclaw，MIT 协议开源，文档质量在我见过的开源项目里算是顶级。从 onboard 命令开始，然后去读 SOUL.md——那里才是真正的入口。
